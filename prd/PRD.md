# PRD: Сервис обработки заявок и определения похожих проектов

**Версия:** 1.0
**Дата:** 2026-03-14
**Статус:** Черновик

---

## 1. Обзор продукта и цели

### 1.1 Контекст

Центр проектной деятельности (ЦПД) ВУЗа ежегодно принимает ~1500 заявок на проекты от преподавателей, партнёров и студентов. Сейчас обработка ведётся в Excel: операторы вручную просматривают заявки, ищут дубликаты и похожие проекты, формируют итоговый список на семестр. Процесс трудоёмкий, непрозрачный и подвержен ошибкам.

### 1.2 Цель продукта

Веб-приложение для операторов ЦПД, которое:

- Хранит заявки на проекты в структурированной базе данных
- Автоматически определяет семантически похожие проекты с помощью ML-модели эмбеддингов
- Предоставляет удобный интерфейс для ручной и автоматической группировки проектов
- Поддерживает процесс отбора проектов на семестр с финальной проверкой на схожесть
- Обеспечивает импорт/экспорт данных из/в Excel

### 1.3 Ключевые метрики успеха

- Сокращение времени на поиск похожих проектов с нескольких дней до минут
- 100% заявок проходят автоматическую проверку на схожесть
- Операторы подтверждают ≥70% автоматических рекомендаций как релевантные

---

## 2. Целевая аудитория

**Основные пользователи:** операторы ЦПД (3–5 человек одновременно). Роль — просмотр заявок, группировка похожих проектов, отбор на семестр.

**Авторизация:** не предусмотрена в текущей версии. Приложение доступно всем, кто имеет к нему сетевой доступ.

## 3. Основные функции

### 3.1 Управление проектами

#### F-01: Просмотр списка проектов

**Описание:** Основной экран приложения — табличный список всех проектов текущего цикла.

**Отображаемые поля в списке (компактный вид):**
- Название проекта
- Проектное направление
- Срок реализации
- Групповая метка (цветной тег, если проект входит в группу)
- Score похожести (мелкий текст у метки группы)

**Счётчики в верхней панели:**
- Всего проектов
- Новых (не проверенных авто-группировкой)
- Авто-проверенных с последнего запуска

**Acceptance criteria:**
- Список загружается за ≤2 секунды при 1500 проектах
- Счётчики обновляются при любом изменении данных
- При клике на проект открывается боковая панель с полными данными

#### F-02: Карточка проекта (боковая панель)

**Описание:** Детальный просмотр всех полей проекта.

**Содержимое:**
- Все поля заявки (см. раздел 5 — модель данных)
- Информация о группе: название, тип (авто/ручная), score, список похожих проектов из группы
- Действия: добавить в группу, убрать из группы, добавить в отбор на семестр

**Acceptance criteria:**
- Панель открывается без перезагрузки страницы (slide-in)
- Все текстовые поля отображаются полностью (без обрезки)

#### F-03: Поиск и фильтрация

**Фильтры:**
- Проектное направление (справочник)
- Приоритетное направление развития ВУЗа (справочник)
- Статус группировки: все / в группе / без группы
- Тип группы: авто / ручная / все
- Уровень готовности технологии (УГТ, справочник)
- Проект уже реализуется: да / нет

**Поиск:** полнотекстовый по полям: название, актуальность, проблема, цель, ключевые задачи, ожидаемый результат.

**Acceptance criteria:**
- Фильтры комбинируются (AND-логика)
- Поиск работает за ≤500 мс
- Фильтры и поиск сохраняются при навигации внутри приложения
- Результаты обновляются при изменении фильтров без перезагрузки

### 3.2 Автоматическая группировка (семантическая похожесть)

#### F-04: Запуск авто-группировки

**Описание:** Кнопка «Найти похожие» в верхней панели запускает процесс семантической классификации.

**Алгоритм:**

1. Для каждого «свободного» проекта (не входящего в ручную или подтверждённую авто-группу) формируется текст: конкатенация полей `название + проблема + цель + ожидаемый результат` (поля с наибольшей смысловой нагрузкой; «актуальность» и «ключевые задачи» исключаются из-за высокой доли шаблонных фраз)
2. Текст нормализуется: удаление стоп-слов, приведение к нижнему регистру
3. Вычисляется эмбеддинг через модель `cointegrated/rubert-tiny2` (локальная, 312-мерный вектор)
4. Вычисляется матрица попарного косинусного сходства
5. Пары со сходством выше порога объединяются в авто-группы (алгоритм: agglomerative clustering с порогом расстояния)
6. Результат сохраняется. Непроверенные авто-группы предыдущего запуска заменяются новыми

**Ограничение:** один проект может входить только в одну группу.

**Поведение при повторном запуске:**
- Ручные группы и подтверждённые авто-группы НЕ затрагиваются
- Работает только с «свободными» проектами и непроверенными авто-группами
- Старые непроверенные авто-группы заменяются новыми

**Acceptance criteria:**
- Обработка 1500 проектов завершается за ≤60 секунд
- Во время обработки отображается прогресс-бар
- После завершения: обновление счётчиков, визуальное обновление списка
- При ошибке — понятное сообщение пользователю

#### F-05: Настройка порога сходства

**Описание:** Оператор может выбрать чувствительность авто-группировки.

**Варианты (пресеты):**
- Строгий (0.85+) — только почти дубликаты
- Средний (0.75+) — баланс
- Мягкий (0.65+) — больше кандидатов, больше шума

**Также:** ручной ввод произвольного значения (0.50–0.95, шаг 0.01).

**Acceptance criteria:**
- Порог задаётся перед запуском авто-группировки
- Значение по умолчанию: 0.75 (средний)
- Последнее выбранное значение сохраняется между сессиями

### 3.3 Ручная группировка

#### F-06: Создание ручной группы

**Описание:** Оператор выделяет несколько проектов в списке (чекбоксы) и создаёт из них группу.

**Поля группы:**
- `name` (string, обязательное) — название группы
- `description` (text, опциональное) — почему объединили
- `source` = "manual"

**Acceptance criteria:**
- Минимум 2 проекта для создания группы
- Нельзя добавить проект, уже входящий в другую группу (показать предупреждение с предложением переместить)
- Группа появляется в списке мгновенно

#### F-07: Редактирование и удаление групп

**Действия:**
- Переименовать группу, изменить описание
- Добавить проект в существующую группу
- Убрать проект из группы (проект становится «свободным»)
- Удалить группу целиком (все проекты становятся «свободными»)
- Подтвердить авто-группу (превращается в зафиксированную, не перезаписывается при повторном запуске)

**Acceptance criteria:**
- Все операции — без перезагрузки страницы
- Подтверждение авто-группы необратимо (но группу можно удалить)

### 3.4 Режимы просмотра

#### F-08: Два режима отображения

**Режим «Список»:** плоский список всех проектов. Групповая принадлежность показана цветной меткой-тегом слева от названия. Проекты одной группы имеют одинаковый цвет. Переключение между режимами — табы в верхней части.

**Режим «Группы»:** аккордеон. Каждая группа — раскрывающийся блок:
- Заголовок: название группы, количество проектов, средний score, тип (авто/ручная), статус (проверена / не проверена)
- Внутри: проекты группы
- Отдельно внизу: секция «Без группы»

**Acceptance criteria:**
- Переключение режимов мгновенное (без запроса к серверу, данные уже загружены)
- Текущий режим сохраняется между сессиями
- Фильтры и поиск работают в обоих режимах

### 3.5 Сравнение проектов

#### F-09: Сравнение двух проектов «лоб в лоб»

**Описание:** Оператор выбирает два проекта и видит их поля рядом в двухколоночном виде.

**Отображение:**
- Поля проектов — параллельно
- Score сходства между ними
- Подсвеченные совпадающие ключевые слова (извлекаются из текста: пересечение значимых слов после лемматизации)

**Acceptance criteria:**
- Доступно из контекстного меню при выборе 2 проектов
- Открывается в модальном окне или отдельной панели

### 3.6 Отбор на семестр

#### F-10: Вкладка «Отбор на семестр»

**Описание:** Отдельная вкладка приложения. Проекты добавляются сюда из основного списка.

**Добавление:** кнопка «Добавить в отбор» в карточке проекта или массовое действие (чекбоксы + кнопка).

**Содержимое вкладки:**
- Компактный список отобранных проектов (те же фильтры и поиск)
- Счётчик отобранных проектов
- Кнопка «Проверить отобранные на схожесть» — запускает авто-группировку только среди отобранных проектов (изолированный контекст, не влияет на группы в основном списке)
- Результат проверки отображается здесь же в режиме групп

**Действия:**
- Убрать проект из отбора
- Экспортировать отобранные в Excel

**Acceptance criteria:**
- Один проект может быть одновременно в группе (в основном списке) и в отборе
- Проверка на схожесть среди отобранных — независимая от основной группировки
- Проверка 300 проектов — ≤15 секунд

### 3.7 Импорт / Экспорт

#### F-11: Импорт из Excel

**Описание:** Загрузка проектов из .xlsx файла.

**Процесс:**
1. Пользователь загружает файл
2. Система валидирует: соответствие колонок полям, заполненность обязательных полей, уникальность названий, корректность значений справочников
3. Показывается превью: сколько проектов будет импортировано, сколько ошибок
4. Пользователь подтверждает импорт
5. Для импортированных проектов автоматически вычисляются эмбеддинги

**Acceptance criteria:**
- Шаблон Excel доступен для скачивания
- Ошибки показываются поимённо: «Строка 47: не заполнено поле "Цель"»
- Дубликаты по названию — предупреждение, не блокировка
- Импорт 1500 строк — ≤30 секунд (без вычисления эмбеддингов; эмбеддинги — фоновая задача)

#### F-12: Экспорт в Excel

**Описание:** Выгрузка проектов в .xlsx файл.

**Варианты:**
- Все проекты
- Только отфильтрованные (текущий набор фильтров)
- Только отобранные на семестр
- Только проекты в группах (с указанием группы и score)

**Acceptance criteria:**
- Экспорт включает все поля проекта + колонку «Группа» + колонку «Score»
- Файл скачивается автоматически
- Экспорт 1500 строк — ≤10 секунд

### 3.8 История авто-группировок

#### F-13: Лог запусков

**Описание:** Раздел с историей всех запусков авто-группировки.

**Сохраняемые данные:**
- Дата и время запуска
- Порог сходства
- Количество обработанных проектов
- Количество найденных групп
- Количество проектов в группах
- Контекст: основной список или отбор на семестр

**Acceptance criteria:**
- История хранится бессрочно
- Доступна для просмотра, не для редактирования

### 3.9 Обратная связь по рекомендациям

#### F-14: Отклонение рекомендаций

**Описание:** Когда оператор удаляет проект из авто-группы или удаляет авто-группу, система сохраняет это как «отклонённую рекомендацию».

**Использование:**
- При следующем запуске авто-группировки отклонённые пары не предлагаются повторно (в рамках текущего цикла)
- Статистика: процент подтверждённых vs отклонённых рекомендаций (доступна в истории)

**Acceptance criteria:**
- Отклонение сохраняется автоматически при удалении из авто-группы
- Не блокирует ручное объединение тех же проектов

### 3.10 Управление справочниками

#### F-15: Админ-панель справочников

**Описание:** Раздел для управления значениями справочников.

**Справочники:**
- Проектные направления (социальные технологии, ИТ и т.д.)
- Приоритетные направления развития ВУЗа
- Уровни готовности технологии (УГТ)

**Действия:** добавить значение, переименовать, деактивировать (не удалять — чтобы не ломать архивные проекты).

**Acceptance criteria:**
- Деактивированные значения не показываются при создании/импорте, но видны в существующих проектах
- Изменения вступают в силу мгновенно

## 5. Концептуальная модель данных

### 5.1 Таблица `projects`

| Поле | Тип | Ограничения | Описание |
|---|---|---|---|
| id | UUID | PK | Уникальный идентификатор |
| title | VARCHAR(80) | NOT NULL | Название проекта |
| direction_id | FK → directions | NOT NULL | Проектное направление |
| is_ongoing | BOOLEAN | NOT NULL | Проект уже реализуется |
| priority_direction_id | FK → priority_directions | NOT NULL | Приоритетное направление ВУЗа |
| duration | VARCHAR(100) | NOT NULL | Срок реализации |
| relevance | TEXT | NOT NULL, max 1500 chars | Актуальность |
| problem | TEXT | NOT NULL, max 1500 chars | Проблема |
| goal | TEXT | NOT NULL, max 1000 chars | Цель |
| key_tasks | TEXT | NOT NULL, max 1500 chars | Ключевые задачи |
| expected_result | TEXT | NOT NULL, max 1000 chars | Ожидаемый продуктовый результат |
| trl_id | FK → trl_levels | NOT NULL | Уровень готовности технологии |
| budget | DECIMAL(12,2) | NOT NULL | Предварительный бюджет, руб. |
| non_financial_support | JSONB | NULLABLE | Нефинансовая поддержка (структурированные чекбоксы) |
| non_financial_support_details | TEXT | NULLABLE, max 3000 chars | Уточнение нефинансовой поддержки |
| team_size | INTEGER | NOT NULL | Планируемое кол-во участников |
| embedding | VECTOR(312) | NULLABLE | Эмбеддинг (pgvector) |
| group_id | FK → groups | NULLABLE | Принадлежность к группе |
| is_selected | BOOLEAN | DEFAULT false | Отобран на семестр |
| is_auto_checked | BOOLEAN | DEFAULT false | Обработан авто-группировкой |
| cycle | VARCHAR(20) | NOT NULL | Цикл (напр. "2026-осень") |
| created_at | TIMESTAMP | NOT NULL | Дата создания |
| updated_at | TIMESTAMP | NOT NULL | Дата обновления |

### 5.2 Таблица `groups`

| Поле | Тип | Ограничения | Описание |
|---|---|---|---|
| id | UUID | PK | Уникальный идентификатор |
| name | VARCHAR(200) | NOT NULL | Название группы |
| description | TEXT | NULLABLE | Описание (почему объединили) |
| source | ENUM('auto','manual') | NOT NULL | Тип: авто или ручная |
| is_confirmed | BOOLEAN | DEFAULT false | Подтверждена оператором (для авто) |
| avg_score | FLOAT | NULLABLE | Средний score внутри группы |
| context | ENUM('main','selection') | NOT NULL | Контекст: основной список или отбор |
| created_at | TIMESTAMP | NOT NULL | Дата создания |

### 5.3 Таблица `similarity_scores`

| Поле | Тип | Ограничения | Описание |
|---|---|---|---|
| id | UUID | PK | Уникальный идентификатор |
| project_a_id | FK → projects | NOT NULL | Первый проект |
| project_b_id | FK → projects | NOT NULL | Второй проект |
| score | FLOAT | NOT NULL | Косинусное сходство (0–1) |
| group_id | FK → groups | NULLABLE | В какой группе эта пара |

UNIQUE constraint: (project_a_id, project_b_id) — одна запись на пару.

### 5.4 Таблица `rejected_pairs`

| Поле | Тип | Ограничения | Описание |
|---|---|---|---|
| id | UUID | PK | Уникальный идентификатор |
| project_a_id | FK → projects | NOT NULL | Первый проект |
| project_b_id | FK → projects | NOT NULL | Второй проект |
| cycle | VARCHAR(20) | NOT NULL | Цикл, в рамках которого отклонено |
| created_at | TIMESTAMP | NOT NULL | Когда отклонено |

### 5.5 Таблица `grouping_runs`

| Поле | Тип | Ограничения | Описание |
|---|---|---|---|

## 6. API-эндпоинты (основные)

### Projects
- `GET /api/projects` — список с фильтрацией, поиском, пагинацией
- `GET /api/projects/{id}` — детали проекта
- `POST /api/projects/import` — импорт из Excel
- `GET /api/projects/export` — экспорт в Excel (query params для фильтров)
- `POST /api/projects/{id}/select` — добавить в отбор
- `DELETE /api/projects/{id}/select` — убрать из отбора
- `GET /api/projects/compare?a={id}&b={id}` — сравнение двух проектов

### Groups
- `GET /api/groups` — список групп (с фильтрацией по source, context)
- `POST /api/groups` — создать ручную группу
- `PATCH /api/groups/{id}` — редактировать группу
- `DELETE /api/groups/{id}` — удалить группу
- `POST /api/groups/{id}/confirm` — подтвердить авто-группу
- `POST /api/groups/{id}/projects` — добавить проект в группу
- `DELETE /api/groups/{id}/projects/{project_id}` — убрать проект из группы

### Grouping
- `POST /api/grouping/run` — запустить авто-группировку (body: threshold, context)
- `GET /api/grouping/status/{run_id}` — статус текущего запуска
- `GET /api/grouping/history` — история запусков

### Dictionaries
- `GET /api/dictionaries/{type}` — значения справочника
- `POST /api/dictionaries/{type}` — добавить значение
- `PATCH /api/dictionaries/{type}/{id}` — редактировать
- `DELETE /api/dictionaries/{type}/{id}` — деактивировать

### Stats
- `GET /api/stats/counters` — счётчики (всего, новых, проверенных)

## 7. Принципы UI-дизайна

### 7.1 Навигация

Четыре вкладки верхнего уровня:
1. **Проекты** — основной экран (список, фильтры, группировка)
2. **Отбор на семестр** — отобранные проекты, финальная проверка
3. **История** — лог запусков авто-группировки
4. **Справочники** — управление значениями справочников

### 7.2 Принципы минимализма

- В списке — минимум полей (4–5 колонок). Детали — только по клику
- Кнопки действий — контекстные: появляются при выделении проектов (массовые операции)
- Статусы и score — ненавязчиво: цветные метки и мелкий текст, не отдельные колонки
- Одно основное действие на экране: на «Проектах» — группировка, на «Отборе» — финализация

### 7.3 Цветовая кодировка групп

- Каждая группа получает цвет из палитры (8–10 различимых цветов)
- Если групп больше палитры — цвета циклически повторяются, но группы различаются по названию
- Авто-группы (неподтверждённые) — пунктирная рамка метки
- Ручные и подтверждённые — сплошная рамка

## 8. Соображения по безопасности

При отсутствии авторизации:

- Приложение должно быть доступно только из внутренней сети ВУЗа или через VPN
- Nginx — ограничение по IP-адресам
- HTTPS обязателен (Let's Encrypt)
- Регулярные бэкапы PostgreSQL (ежедневно, хранение 30 дней)
- Rate limiting на API (защита от случайных массовых запросов)

**Рекомендация на будущее:** при расширении числа пользователей или выходе в публичную сеть — добавить авторизацию (OAuth2 / LDAP интеграция с ВУЗовской системой).

## 9. Этапы разработки

### Этап 1: Основа (3–4 недели)

- Настройка инфраструктуры: Docker Compose (PostgreSQL, Redis, FastAPI, Vue dev server)
- Модель данных, миграции (Alembic)
- CRUD API для проектов
- Импорт из Excel с валидацией
- Фронтенд: список проектов, карточка, поиск, фильтры

**Результат:** можно загрузить проекты и просматривать их.

### Этап 2: ML-группировка (2–3 недели)

- Интеграция rubert-tiny2 (sentence-transformers)
- Celery worker для вычисления эмбеддингов
- Алгоритм авто-группировки (agglomerative clustering)
- API для запуска, статуса, результатов
- Фронтенд: кнопка «Найти похожие», прогресс, отображение авто-групп
- Настройка порога сходства

**Результат:** автоматический поиск похожих проектов работает.

### Этап 3: Ручная группировка и сравнение (1–2 недели)

- CRUD для ручных групп
- Подтверждение/отклонение авто-групп
- Режимы просмотра (список / группы)
- Сравнение двух проектов
- Учёт отклонённых рекомендаций

**Результат:** полный цикл группировки (авто + ручная).

### Этап 4: Отбор и финализация (1–2 недели)

- Вкладка «Отбор на семестр»
- Добавление/удаление из отбора
- Проверка отобранных на схожесть (изолированный контекст)
- Экспорт в Excel (все варианты)

**Результат:** полный рабочий процесс от импорта до финального списка.

### Этап 5: Полировка (1 неделя)

- История запусков
- Управление справочниками
- Статистика (процент подтверждённых рекомендаций)
- Оптимизация производительности
- Тестирование на реальных данных

**Итого: 8–12 недель** при команде 1–2 разработчика.

## 10. Потенциальные проблемы и решения

| Проблема | Решение |
|---|---|
| «Водянистые» тексты завышают сходство | Предобработка: удаление шаблонных фраз. Использовать только содержательные поля (название + проблема + цель + результат) |
| Разная терминология для одного и того же | Эмбеддинговая модель частично решает. Дополнительно: учитывать направление как мета-фильтр |
| 1500 проектов — матрица 1500×1500 | ~1.1M пар. При 312-мерных векторах — несколько секунд на CPU с numpy. Не проблема |
| Модель занимает RAM при простое | Загружать модель в Celery worker, не в основной процесс FastAPI. Worker можно перезапускать |
| Потеря данных | Ежедневные бэкапы PostgreSQL. Docker volumes для персистентности |
| Одновременная работа 3–5 операторов | FastAPI + async — достаточно. Конфликты при одновременной группировке: запускать только один процесс авто-группировки за раз (блокировка через Redis) |
